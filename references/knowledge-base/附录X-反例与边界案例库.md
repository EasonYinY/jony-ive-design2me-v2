## 0. 为什么需要反例库

```
┌─────────────────────────────────────────────────────────────────────┐
│                    知识的两面                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  成功案例告诉我们"应该怎么做"                                    │

```

## 1. "Apple-like" 误用识别

### 1.1 什么是"Apple-like"误用

```
┌─────────────────────────────────────────────────────────────────────┐
│                    典型误用：模仿Apple外观                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【错误理解】                                                     │
│  Apple的设计 = 白色 + 圆角 + 极简                                 │
│  所以我也这样做 = Apple的设计                                     │
│                                                                     │
│  【为什么错】                                                     │
│  → 模仿了形态，没有理解关系                                       │
│  → Apple的克制 = 减少选择                                        │
│  → 模仿Apple外观 = 反而增加了错误的选择                           │
│  → Apple方法是"关系正确"，模仿只是"形态相似"                    │
│                                                                     │
│  【艾维怎么说】                                                   │
│  "我们不是在追求一种风格，我们在追求正确"                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 误用案例对比

| 案例 | 错误做法 | 问题 | 正确做法 |
|------|----------|------|----------|
| **某智能音箱** | 模仿 iPod 白色极简风格 | 模仿形态，不理解关系 | 问：这个产品要建立什么关系？ |
| **某电动车** | 模仿 Tesla 极简内饰 | 没有理解触觉优先 | 问：用户的手需要确认什么？ |
| **某App图标** | 模仿 Apple 圆角卡片 | 圆角不是目的 | 问：这个图标的触觉确认是什么？ |
| **某网页** | 大量留白 + 衬线字体 | 留白不是克制 | 问：什么需要让位？ |

### 1.3 "Apple-like" 修正流程

```
当发现"Apple-like"误用时：

Step 1: 识别误用信号
├── 大量使用白色
├── 追求圆角
├── 追求无缝
├── 追求极薄
└── 使用"Apple设计"作为理由

Step 2: 重写关系命题
├── "这个产品不是[品类]"
├── "这是[谁]与[行为]、[环境]、[信任]的关系"
└── 不是台灯，是人与光、携带、熄灭的关系

Step 3: 重新推导形态
├── 形态是否服务于关系？
├── 形态是否克制而非堆砌？
└── 形态是否十年后仍正确？

Step 4: 验证
├── 如果没有 Apple 这个产品还正确吗？
└── 如果答案是"错误"，则是误用
```

## 2. 材料"高级感"误用

### 2.1 什么是材料误用

```
┌─────────────────────────────────────────────────────────────────────┐
│                    典型误用：用"高级"描述材料                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【禁止使用的词】[S]                                               │
│  ❌ premium metal                                                  │
│  ❌ luxury wood                                                   │
│  ❌ crystal clear                                                 │
│  ❌ high-end finish                                               │
│  ❌ elegant texture                                               │
│  ❌ sophisticated                                                  │
│  ❌ premium feel                                                  │
│                                                                     │
│  【为什么错】                                                     │
│  → "高级"无法验证                                                 │
│  → 不同人对"高级"理解不同                                        │
│  → 无法指导设计决策                                               │
│  → 是主观感受，不是客观职责                                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 误用案例对比

| 错误写法 | 为什么错 | 正确写法 |
|----------|----------|----------|
| "采用高级金属材质" | 无法验证，不同人理解不同 | "brushed stainless steel for salt corrosion resistance" |
| "奢华皮革包装" | 质感词，不说功能 | "full-grain leather for patina aging" |
| "水晶般透明" | 比喻词，不说职责 | "optically clear polymer for light transmission" |
| "高端质感玻璃" | 无法验证 | "precision-ground glass for even light diffusion" |
| "优雅的纹理" | 主观感受 | "embossed pattern for grip confirmation" |

### 2.3 材料职责修正流程

```
当发现材料描述问题时：

Step 1: 识别误用信号
├── 使用了禁止词（premium/luxury/crystal/elegant）
├── 没有"for"连接
└── 只描述感受，不描述职责

Step 2: 追问三个问题
├── Q1: 这个材料在这里做什么功能？
├── Q2: 这个材料传达什么情绪？
└── Q3: 这个材料十年后会变成什么样？

Step 3: 改写为职责格式
├── "[材料] for [具体功能]"
├── "[材料] for [情绪品质]"
└── "[材料] for [时间老化]"

Step 4: 验证
├── 能说出为什么选这个材料吗？
└── 如果只能说"好看/高级"，则是误用
```

## 3. "极简" 误用识别

### 3.1 什么是"极简"误用

```
┌─────────────────────────────────────────────────────────────────────┐
│                    极简 = 关系清楚，而非数量少                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【错误理解】                                                     │
│  极简 = 元素越少越好                                              │
│  极简 = 白色 + 留白                                              │
│  极简 = 没有装饰                                                  │
│                                                                     │
│  【正确理解】[S]                                                  │
│  极简 = 关系清楚                                                  │
│  极简 = 每个元素都有职责                                          │
│  极简 = 克制是力量                                                │
│                                                                     │
│  【艾维怎么说】                                                   │
│  "简洁是复杂的最终形式"                                          │
│  "大量思考、工程、协作要沉到安静结果里"                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 误用案例对比

| 错误做法 | 为什么错 | 正确做法 |
|----------|----------|----------|
| 减少元素但关系不清楚 | 元素少≠关系清楚 | 先定义关系，再决定元素 |
| 大量留白但不知道让位什么 | 留白是手段，不是目的 | 问：什么需要让位？ |
| 删除装饰但失去识别度 | 克制不是无特征 | 克制+明确的关系 |
| 追求无缝但失去触觉确认 | 无缝可能破坏功能 | 问：什么需要可见？ |

### 3.3 "极简"修正流程

```
当发现"极简"误用时：

Step 1: 识别误用信号
├── 纯粹为了少而少
├── 没有明确"让位"的对象
└── 失去必要的功能元素

Step 2: 追问关系
├── 这个产品要建立什么关系？
├── 什么需要强调？什么需要让位？
└── 克制是为了什么？

Step 3: 重新定义克制
├── 克制 = 大量思考后的提炼
├── 克制 = 复杂性的升华
└── 克制 = 让核心关系更清楚

Step 4: 验证
├── 能说清楚"少"的原因吗？
└── 如果只是"看起来简洁"，则是误用
```

## 4. "消失型硬件" 边界

### 4.1 什么时候不能消失

```
┌─────────────────────────────────────────────────────────────────────┐
│                    消失的边界                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【可以消失的】                                                    │
│  ├── 对任务的注意力干扰                                           │
│  ├── 不必要的视觉噪音                                             │
│  ├── 过度设计的装饰                                                │
│  └── 技术存在的感知（当技术不服务于体验时）                        │
│                                                                     │
│  【不能消失的】                                                    │
│  ├── 触觉确认节点（高频交互需要身体确认）                         │
│  ├── 品牌识别（需要可识别性）                                     │
│  ├── 安全相关元素（不能为了美而牺牲安全）                         │
│  └── 必要的反馈机制                                               │
│                                                                     │
│  【消失的方向】                                                   │
│  消失 = 让出注意力                                                │
│  不是目标，是结果                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 边界案例

| 场景 | 消失方向 | 不能消失 | 原因 |
|------|----------|----------|------|
| AirPods | 对"设备"的注意力 | 充电盒（身份识别） | 品牌需要可见性 |
| Apple Watch | 对"手表"的注意力 | 表冠（高频控制） | 触觉需要确认 |
| 电动车仪表盘 | 对"屏幕"的注意力 | 方向盘握感 | 安全需要触觉 |

### 4.3 消失设计检查

```
消失型设计必须先定义：

Q1: 被让出的是什么？
├── 注意力从什么移开？
└── 为什么需要移开？

Q2: 让给的是什么？
├── 注意力去向哪里？
└── 这个方向更重要吗？

Q3: 留下的最小接口是什么？
├── 什么必须保留？
└── 为什么必须保留？

禁止：
❌ 只写 invisible / seamless / minimal
✓ 必须回答：让给什么？留下什么？
```

## 5. "熟悉非怀旧" 边界

### 5.1 熟悉 vs 怀旧

```
┌─────────────────────────────────────────────────────────────────────┐
│                    熟悉 vs 怀旧的边界                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【熟悉】[S]                                                       │
│  ├── 引用行为/秩序，不引用外形                                    │
│  ├── 用户不需要学习就会用                                          │
│  ├── 用现代材料实现同样的解决方案                                 │
│  └── 引用 archetype/ritual/instrument                              │
│                                                                     │
│  【怀旧】                                                          │
│  ├── 复制旧时代的外观                                             │
│  ├── 假装技术停在过去                                              │
│  ├── 没有理解为什么过去的东西是好的                               │
│  └── 只引用外形，不引用本质                                        │
│                                                                     │
│  【必须附】[S]                                                     │
│  └─ not nostalgic replica                                         │
│  └─ not retro prop                                                │
│  └─ not decorative pastiche                                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 边界案例

| 案例 | 熟悉 | 怀旧 |
|------|------|------|
| Sailing Lantern | 圆形+玻璃+金属=航海秩序 | 不用1950年代材料 |
| Ferrari Luce | 四圆灯=品牌仪式 | EV架构是现代的 |
| LP12-50 | 唱盘架构=聆听仪式 | 金属铰链是新的 |

### 5.3 熟悉度检查

```
检查这个引用是否越界：

□ 是否引用了动作/秩序，而非外形？
□ 是否有 not nostalgic replica 约束？
□ 是否用现代材料实现？
□ 引用是否服务于当前关系命题？
□ 还是只是在复制旧时代符号？

如果是 → 越界了
```

## 6. "协作是设计对象" 边界

### 6.1 什么不是协作设计

```
┌─────────────────────────────────────────────────────────────────────┐
│                    外包 vs 协作的边界                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【外包】❌                                                        │
│  ├── 发 brief → 等结果                                            │
│  ├── 给出方案，拿走就走                                            │
│  ├── 能力留在设计方                                                │
│  ├── 合作方依赖设计方                                              │
│  └── 结束 = 一切停止                                              │
│                                                                     │
│  【协作】✓                                                        │
│  ├── 进入系统 → 共同创造                                          │
│  ├── 帮建能力，而非替做                                            │
│  ├── 能力转移给合作方                                              │
│  ├── 合作方在结束后更强                                            │
│  └── 结束 = 新开始                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 边界判断

| 判断问题 | 外包 | 协作 |
|----------|------|------|
| 谁做决策？ | 设计方 | 共同 |
| 能力在哪里？ | 设计方 | 双方 |
| 结束后谁更强？ | 设计方 | 合作方 |
| 合作方能独立吗？ | 不能 | 能 |

## 7. 通用误用检测清单

```
在每次设计决策后，用这个清单检查：

【关系命题】
□ 能用一句话说清楚这是什么关系吗？
□ 形态是否服务于关系？
□ 是否有"Apple-like"模仿？

【材料】
□ 是否使用了禁止词？
□ 每个材料能回答"for什么"吗？
□ 材料有功能/情绪/时间职责吗？

【触觉】
□ 有3个以上触觉确认节点吗？
□ 每个节点能回答四问题吗？
□ 是否有"无缝"破坏了功能？

【克制】
□ 知道什么在"让位"吗？
□ 少是有意为之，还是盲目追求？
□ 克制后关系更清楚了吗？

【边界】
□ 什么情况下这个方法失效？
□ 是否有反例被忽视？
□ 是否机械套用了成功案例？

【协作】（如适用）
□ 是在进入系统还是外包？
□ 能力转移了吗？
□ 结束后合作方能独立吗？
```
