## 0. 迁移概述

### 0.1 为什么需要迁移指南

```
艾维方法论的核心不是"产品设计"
而是"关系命题"的思维方式

关系命题可以应用于：
- 任何涉及人与系统交互的场景
- 任何需要建立信任的设计
- 任何需要长期考虑的系统

方法论迁移 ≠ 方法复制
方法论迁移 = 核心思维的应用
```

### 0.2 迁移的核心原则

```
┌─────────────────────────────────────────────────────────────────────┐
│                    迁移三原则                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【原则1：保留核心】                                             │
│  关系命题 × 材料职责 × 触觉确认 × 时间检查                      │
│  → 这些是方法论的核心，必须保留                                │
│                                                                     │
│  【原则2：转化形式】                                             │
│  物理材料 → 数字元素                                            │
│  触觉节点 → 交互确认                                            │
│  时间老化 → 持续性评估                                          │
│  → 形式要适应新领域                                            │
│                                                                     │
│  【原则3：验证边界】                                             │
│  新领域应用后必须验证                                            │
│  方法论是否真正解决了问题？                                      │
│  边界在哪里？                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```

关系命题 → 服务关系命题
材料职责 → 触点职责

```

```
This is not a [service category];

这不是一个[服务品类]；
这是[客户]与[服务触点]、[交互体验]、[信任]的关系。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【客户】谁在接受服务？
- 行为特征
- 情感需求
- 信任期望

【服务触点】客户在哪里与服务接触？
- 物理触点：门店、员工、设备
- 数字触点：App、网站、客服
- 人工触点：电话、面对面

【交互体验】服务过程中有什么体验？
- 等待体验
- 获取体验
- 问题解决体验
- 结束体验

【信任】服务建立什么信任？
- 能力信任：能解决问题
- 态度信任：被尊重
- 持续信任：长期可靠
```

### 1.4 服务关键时刻

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
服务关键时刻设计
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

服务中的"触觉节点" = 关键时刻

【关键时刻识别】
- 客户会反复确认什么？
- 什么时刻影响信任？
- 什么时刻决定体验？

【关键时刻设计】
- 确认点的反馈是什么？
- 反馈形式：语言/动作/环境/数字？
- 如何让客户感到"被确认"？

例：酒店入住
关键时刻：前台办理
确认点：接待语言、身份确认速度、入住卡交付
反馈：欢迎语+效率+房间钥匙的位置
```

### 1.5 服务时间检查

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
服务周期 · 时间检查
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【服务稳定性】
- 服务质量是否一致？
- 体验是否可重复？

【服务演变】
- 服务如何随时间进化？
- 新客户vs老客户体验差异？

【服务结束】
- 服务如何优雅结束？
- 结束体验是否被考虑？

【服务退出】
- 客户如何离开服务？
- 退出体验是否被设计？
```

### 1.6 服务设计案例

```
案例：Airbnb 民宿入住

关系命题：
"This is not a hotel check-in;
it is a relationship between a traveler
and welcome, authenticity, and local trust."

服务触点职责：
钥匙交接 for 入住仪式（功能）
for 欢迎氛围（情感）
for 家的感觉（周期）

关键时刻：
门锁密码/钥匙盒 → 第一次物理接触
欢迎包 → 第一印象确认
房间指南 → 功能熟悉

时间检查：
入住体验一致性 → 新老房东体验是否同样温暖？
紧急响应 → 问题能否快速解决？
```
