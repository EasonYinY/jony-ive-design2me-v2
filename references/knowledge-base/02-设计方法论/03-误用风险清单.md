# 02-补充. 误用风险清单

> **版本**：1.0
> **日期**：2026-06-22
> **定位**：显性化16法应用的常见误用，建立风险识别和修正体系
> **目标**：防止方法论被机械套用或误解

## 1. 方法1：语言先行的误用风险

### 1.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：把动词表当成装饰                                        │
│  ──────────────────────────────────────────────────────────────── │
│  表现：列出 extinguish/carry/hang 但形态设计与动词无关           │
│                                                                    │
│  问题：动词只是词汇，没有转化为设计决策                          │
│                                                                    │
│  修正：动词必须推导出形态                                        │
│  → extinguish → 需要熄灭动作确认                                  │
│  → carry → 需要携带结构                                          │
│  → hang → 需要悬挂点                                            │
│                                                                    │
│  误用2：用中文翻译替代英文精确动词                               │
│  ──────────────────────────────────────────────────────────────── │
│  表现：用"关灯"替代"extinguish"                                 │
│                                                                    │
│  问题：中文动词不够精确，无法指导形态设计                        │
│                                                                    │
│  修正：必须使用英文精确动词或创造新的精确描述                    │
│  → "熄灭"而非"关"                                              │
│  → "调校"而非"调节"                                            │
│  → "携带"而非"拿"                                              │
│                                                                    │
│  误用3：只换动词，不理解行为差异                                │
│  ──────────────────────────────────────────────────────────────── │
│  表现：知道"extinguish vs switch off"但不知道为什么              │
│                                                                    │
│  问题：知道不同，但不理解为什么不同                              │
│                                                                    │
│  修正：理解每个动词背后的行为逻辑                                │
│  → extinguish = 有仪式感的熄灭 → 需要旋钮动作                    │
│  → switch off = 电子关断 → 可能只是按钮                        │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 1.2 风险识别信号

```
当出现以下信号时，可能是语言先行误用：

□ 行为动词列出了，但没有对应的形态决策
□ 用中文动词替代英文，没有创造新的精确描述
□ 动词之间的逻辑关系不清晰
□ 形态设计无法回溯到任何一个行为动词
□ 动词选择随意，可以被其他动词替代而不影响设计
```

### 1.3 修正流程

```
Step 1: 检查动词与形态的对应关系
├── 每个动词都有对应的形态决策吗？
└── 如果没有 → 重写

Step 2: 检查动词的精确性
├── 动词是否足够精确？
├── 能否被其他动词替代？
└── 如果可以 → 寻找更精确的动词

Step 3: 检查动词之间的逻辑
├── 动词之间的关系是什么？
├── 是否有主次之分？
└── 形态是否反映了这种关系？

Step 4: 验证
└── 从形态能回溯到动词吗？
```

## 2. 方法2：先重写关系的误用风险

### 2.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：重写关系但形态未变                                       │
│  ──────────────────────────────────────────────────────────────── │
│  表现：写出了漂亮的关系命题，但设计形态与传统品类无异              │
│                                                                    │
│  问题：关系命题只是装饰，没有指导设计                            │
│                                                                    │
│  修正：关系命题必须改变设计决策                                  │
│  → "这不是台灯" → 形态必须不是台灯                             │
│  → "这是人与光的关系" → 光的行为决定形态                        │
│                                                                    │
│  误用2：关系命题过于抽象，无法指导设计                          │
│  ──────────────────────────────────────────────────────────────── │
│  表现："这是人与美好生活的关系"                                  │
│                                                                    │
│  问题：太抽象，无法推导出具体设计                                │
│                                                                    │
│  修正：关系命题必须具体到可推导形态                              │
│  → "这是人与专注照明、便携移动、温暖陪伴的关系"                │
│                                                                    │
│  误用3：接受默认品类答案                                         │
│  ──────────────────────────────────────────────────────────────── │
│  表现：一开始就说"设计一个台灯"                                  │
│                                                                    │
│  问题：没有挑战默认答案                                           │
│                                                                    │
│  修正：必须先问"这是什么关系？"                                 │
│  → "这不是台灯，这是..."                                       │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 风险识别信号

```
当出现以下信号时，可能是关系命题误用：

□ 关系命题写得很漂亮，但设计形态与传统品类无异
□ 关系命题过于抽象（如"美好生活"）
□ 一开始就接受了品类答案（如"设计一个台灯"）
□ 无法从关系命题推导出具体设计决策
□ 形态决策与关系命题无关
```

### 2.3 修正流程

```
Step 1: 检查是否挑战了默认品类
├── 是否有"这不是X"的结构？
└── 如果没有 → 重写关系命题

Step 2: 检查关系的具体性
├── 关系命题是否具体到可推导形态？
├── 能否从命题推导出材料、触觉、形态？
└── 如果太抽象 → 细化到可操作层面

Step 3: 检查形态是否反映关系
├── 形态决策能否回溯到关系命题？
├── 如果去掉关系命题，形态会改变吗？
└── 如果不会 → 关系命题只是装饰

Step 4: 验证
└── 这个关系命题改变了什么设计决策？
```

## 3. 方法3：熟悉非怀旧的误用风险

### 3.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：引用了外形而非行为                                       │
│  ──────────────────────────────────────────────────────────────── │
│  表现：用了老式灯罩外形，但行为是现代的                         │
│                                                                    │
│  问题：熟悉感来自行为，不来自外形                                │
│                                                                    │
│  修正：引用行为/秩序，不引用外形                                │
│  → 圆形 = 航海灯的秩序 → 可以引用                              │
│  → 复古灯罩 = 怀旧外形 → 不可引用                             │
│                                                                    │
│  误用2：缺少"not nostalgic replica"约束                        │
│  ──────────────────────────────────────────────────────────────── │
│  表现：引用了archetype但没有明确"不是复制"                     │
│                                                                    │
│  问题：可能变成怀旧复制                                          │
│                                                                    │
│  修正：必须同时包含引用和约束                                    │
│  → "archetype, ritual, but not nostalgic replica"               │
│                                                                    │
│  误用3：理解了但执行不到位                                     │
│  ──────────────────────────────────────────────────────────────── │
│  表现：知道要引用但分寸拿捏不准                                │
│                                                                    │
│  问题：可能过于怀旧或过于现代                                    │
│                                                                    │
│  修正：明确"熟悉"的是行为，而非形式                            │
│  → 行为相同，材料更新                                           │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 3.2 风险识别信号

```
当出现以下信号时，可能是熟悉非怀旧误用：

□ 引用的是外形符号而非行为/秩序
□ 没有"not nostalgic replica"约束
□ 形态过于接近原型，难以区分
□ 可以被称为"复古产品"
□ 材料选择与原型相同（而非现代材料）
```

### 3.3 修正流程

```
Step 1: 检查引用的是什么
├── 引用的是外形？行为？秩序？
├── 如果是外形 → 改为引用行为/秩序
└── 如果是行为/秩序 → 继续

Step 2: 检查是否有约束
├── 是否有"not nostalgic replica"？
├── 是否有"but not retro prop"？
└── 如果没有 → 添加约束

Step 3: 检查材料选择
├── 是否使用现代材料？
├── 现代材料如何实现同样的行为/秩序？
└── 如果使用传统材料 → 重新考虑

Step 4: 验证
└── 用户看到这个产品会联想到原型吗？
    但它确实不是原型吗？
```

## 4. 方法4：边界即尊重的误用风险

### 4.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：把边界当限制                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：抱怨边界"限制了创意"                                      │
│                                                                    │
│  问题：边界是设计输入，不是限制                                  │
│                                                                    │
│  修正：把边界当作设计方向的线索                                  │
│  → "这个边界在告诉我什么？"                                      │
│                                                                    │
│  误用2：忽视某些边界                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：只考虑技术边界，忽视维护边界                             │
│                                                                    │
│  问题：设计出来但无法维护                                        │
│                                                                    │
│  修正：全面识别所有相关边界                                      │
│  → 环境边界、法规边界、维护边界、公开事实                       │
│                                                                    │
│  误用3：把边界绝对化                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：边界神圣不可侵犯                                          │
│                                                                    │
│  问题：有些边界可以被挑战，有些不能                              │
│                                                                    │
│  修正：区分可谈判边界和不可谈判边界                              │
│  → 法规 = 不可谈判                                              │
│  → 技术限制 = 可谈判（等待技术进步）                            │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 4.2 风险识别信号

```
当出现以下信号时，可能是边界处理误用：

□ 抱怨边界"限制了创意"
□ 遗漏了某些边界类型
□ 把所有边界都当作不可谈判
□ 边界没有反映在设计决策中
□ 无法解释设计如何回应每个边界
```

### 4.3 修正流程

```
Step 1: 全面识别边界
├── 环境边界
├── 法规边界
├── 材料边界
├── 维护边界
└── 公开事实

Step 2: 区分边界类型
├── 不可谈判边界
└── 可谈判边界

Step 3: 回应每个边界
├── 这个边界在告诉我什么？
├── 设计如何回应这个边界？
└── 边界如何成为设计决策的依据？

Step 4: 验证
└── 设计决策能对应到每个边界吗？
```

## 5. 方法5：材料诚实的误用风险

### 5.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：使用了禁止词                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：premium metal / luxury wood / crystal clear              │
│                                                                    │
│  问题：这些词无法验证，缺乏职责                                  │
│                                                                    │
│  修正：使用"for what"格式                                       │
│  → "brushed stainless steel for corrosion resistance"            │
│                                                                    │
│  误用2：写了职责但无法验证                                       │
│  ──────────────────────────────────────────────────────────────── │
│  表现："木材传达温暖感"                                         │
│                                                                    │
│  问题：温暖感无法验证，不同人理解不同                            │
│                                                                    │
│  修正：使用可验证的描述                                          │
│  → "solid oak for structural warmth"                             │
│  → "for domestic comfort, verified by user studies"             │
│                                                                    │
│  误用3：缺少时间维度                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：只写了功能和情绪，忽略时间                               │
│                                                                    │
│  问题：不完整，无法评估长期效果                                    │
│                                                                    │
│  修正：每个材料必须回答三个问题                                  │
│  → 功能 + 情绪 + 时间老化                                       │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 5.2 风险识别信号

```
当出现以下信号时，可能是材料诚实误用：

□ 使用了禁止词（premium/luxury/crystal/elegant）
□ 材料描述无法验证（"温暖感"）
□ 材料只有功能描述，没有情绪和时间维度
□ 材料选择随意，无法解释为什么选这个
□ 材料之间没有逻辑关系
```

### 5.3 修正流程

```
Step 1: 检查禁止词
├── 是否有premium/luxury/crystal/elegant？
├── 如果有 → 替换为"for what"格式
└── 继续

Step 2: 检查职责
├── 每个材料能回答"for what"吗？
├── 描述可验证吗？
└── 如果不能 → 重写

Step 3: 检查完整性
├── 功能维度？情绪维度？时间维度？
├── 三个维度都完整吗？
└── 如果不完整 → 补充

Step 4: 验证
└── 每个材料能回答三个问题吗？
    功能是什么？情绪是什么？十年后会怎样？
```

## 6. 方法6：触觉优先的误用风险

### 6.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：把装饰当触觉                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：纹理是装饰性的，没有承担行为                              │
│                                                                    │
│  问题：触觉节点必须承担行为，不是装饰                            │
│                                                                    │
│  修正：每个触觉节点必须回答四问题                                │
│  → 确认什么行为？反馈是什么？位置？频率？                      │
│                                                                    │
│  误用2：触觉节点太少                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：只有1-2个触觉节点                                        │
│                                                                    │
│  问题：高频交互无法确认                                          │
│                                                                    │
│  修正：确保有3个以上触觉确认点                                  │
│                                                                    │
│  误用3：高频交互用触屏                                           │
│  ──────────────────────────────────────────────────────────────── │
│  表现：亮度调节用触屏滑动                                       │
│                                                                    │
│  问题：触屏无法提供物理反馈                                      │
│                                                                    │
│  修正：高频交互用物理控制                                        │
│  → "旋钮 vs 触屏滑动"                                          │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 6.2 风险识别信号

```
当出现以下信号时，可能是触觉优先误用：

□ 触觉元素是装饰性的，没有承担行为
□ 触觉节点少于3个
□ 高频交互用触屏而非物理控制
□ 无法回答触觉节点的四问题
□ 触觉反馈不清晰
```

### 6.3 修正流程

```
Step 1: 识别高频交互
├── 用户会反复做什么？
├── 什么操作需要确认？
└── 如果高频交互用触屏 → 需要改为物理控制

Step 2: 添加触觉节点
├── 识别所有触觉确认点
├── 确保有3个以上
└── 每个节点回答四问题

Step 3: 定义反馈形式
├── 阻尼？声音？位置？状态点？
└── 反馈足够清晰吗？

Step 4: 验证
└── 用户的手能在3个以上位置确认状态吗？
```

## 7. 方法7：维护是生命的误用风险

### 7.1 常见误用模式

```
┌────────────────────────────────────────────────────────────────────┐
│                    误用模式                                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  误用1：把可维修性当风格                                        │
│  ──────────────────────────────────────────────────────────────── │
│  表现：螺丝外露是为了"好看"                                     │
│                                                                    │
│  问题：可见螺丝是为了可维修，不是装饰                            │
│                                                                    │
│  修正：可维修性是功能，不是风格                                  │
│  → 螺丝可见 → 因为需要被操作                                    │
│                                                                    │
│  误用2：忽视时间维度                                             │
│  ──────────────────────────────────────────────────────────────── │
│  表现：只考虑维修，不考虑老化                                    │
│                                                                    │
│  问题：不完整                                                    │
│                                                                    │
│  修正：完整的时间检查                                            │
│  → 可维修 + 可清洁 + 可老化 + 可更换                           │
│                                                                    │
│  误用3：过度设计维护性                                          │
│  ──────────────────────────────────────────────────────────────── │
│  表现：每个部件都强调可更换                                      │
│                                                                    │
│  问题：有些部件不需要可更换                                       │
│                                                                    │
│  修正：区分需要可更换和不需要可更换的部件                        │
│  → 织带/表带 → 可更换                                          │
│  → 核心框架 → 不需要可更换                                      │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 7.2 风险识别信号

```
当出现以下信号时，可能是维护是生命误用：

□ 可见螺丝/接点但不是为了可维修
□ 没有完整的时间检查（维修/清洁/老化/更换）
□ 过度强调可更换性
□ 无法解释为什么某些部件要可见
□ 时间老化设计不完整
```

### 7.3 修正流程

```
Step 1: 识别关键部件
├── 什么是易耗品？需要更换？
├── 什么是核心？不需要更换？
└── 分开处理

Step 2: 设计维护路径
├── 每个需要维护的部件都有路径吗？
├── 螺丝位置合理吗？
└── 工具通用吗？

Step 3: 设计老化方式
├── 什么材料会老化好看？
├── 老化后功能保持吗？
└── 老化方式与产品定位一致吗？

Step 4: 验证
└── 用户十年后会怎么看这个产品？
```

## 8. 方法8-16：快速检查清单

### 8.1 方法9：秩序收复杂

```
□ 复杂元素是否被秩序控制？
□ 秩序是否服务于关系？
□ 秩序是圆形/网格/排版/层级？
□ 是否解决了混乱问题？
```

### 8.2 方法10：协作是设计对象

```
□ 是进入系统还是外包？
□ 能力转移了吗？
□ 合作方比之前更强了吗？
□ 协作深度合适吗？
```

### 8.3 方法11：反屏幕主义

```
□ 高频交互用物理控制了吗？
□ 屏幕用于信息显示，物理用于操作？
□ 触觉节点是否到位？
□ 分布式界面是否合理？
```

### 8.4 方法12：摄影服务判断

```
□ 四类证据是否齐全？
□ 缺失角度是否标注？
□ 每个角度回答什么问题？
□ 证据是否验证了判断？
```

### 8.5 方法13：模块化身体关系

```
□ 主体和接口是否分离？
□ 分离是否合理？
□ 材料/节奏是否不同？
□ 生命周期是否分开管理？
```

### 8.6 方法14：公共图形规范

```
□ 是否输出了规范？
□ 规范是否包含clear zone？
□ 是否有禁止变形规则？
□ 是否定义了使用场景？
```

### 8.7 方法15：服务与组织

```
□ 产品是设计对象，组织也是设计对象？
□ 合作方团队是第二个设计对象？
□ 是帮合作方成长还是替做？
□ 协作后合作方能独立吗？
```

### 8.8 方法16：消失型硬件

```
□ 定义了被消失的是什么吗？
□ 定义了让给的是什么吗？
□ 定义了留下的最小接口吗？
□ 留下的是否必要？
```

## 9. 通用误用检测流程

```
当发现设计有问题时，按以下流程检查：

Step 1: 识别哪个方法可能被误用
├── 根据问题类型判断
└── 参考各方法的误用模式

Step 2: 检查该方法的误用信号
├── 列出该方法的常见误用
├── 检查是否有信号出现
└── 如果有 → 确定误用类型

Step 3: 执行该方法的修正流程
├── 参考各方法的修正步骤
└── 逐项修正

Step 4: 验证修正结果
├── 用该方法的验证标准
└── 确保没有引入新问题

Step 5: 检查对其他方法的影响
├── 修正是否影响其他方法？
└── 需要调整其他方法吗？
```
