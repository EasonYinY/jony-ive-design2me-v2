## 方法1: 语言先行，用词改变产品行为

### 核心原理 [S][A]

```
❌ 错误做法：先想外形，再找描述词
✓ 正确做法：先找行为动词，再生成外形

行为动词示例：
┌────────────────────────────────────────────┐
│                                            │
│  extinguish  vs  switch off                │
│  (熄灭蜡烛/灯，不是简单关灯)                │
│                                            │
│  carry        vs  hold                     │
│  (悬挂/背负，不是仅仅握住)                   │
│                                            │
│  hang         vs  place                    │
│  (悬挂于某处，不是放在桌上)                  │
│                                            │
│  tune         vs  adjust                   │
│  (调校/精调，不是简单调节)                   │
│                                            │
│  unlock      vs  open                     │
│  (解锁开启，不是普通开门)                    │
│                                            │
│  seal         vs  close                    │
│  (封缄/密封，不是简单关闭)                   │
│                                            │
│  listen       vs  play                     │
│  (聆听仪式，不是播放音乐)                    │
│                                            │
│  drive       vs  operate                   │
│  (驾驭仪式，不是操作机器)                    │
│                                            │
└────────────────────────────────────────────┘
```

### 深度理解：为什么"语言先行"？

```
表面理解：
→ 用动词替代名词

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么？
→ 形态是由行为决定的
→ "关灯" vs "熄灭" → 不同的行为 → 不同的形态
→ "拿东西" vs "携带" → 不同的行为 → 不同的结构

本质是什么？
→ 设计不是造型工作，是行为设计
→ 先理解人需要做什么，再决定形态
→ 形态是行为的自然结果

怎么做？
→ 不问"这个产品应该长什么样"
→ 问"人需要做什么行为"
→ 再问"什么形态能最好地实现这些行为"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Ferrari Luce：
- 不是"好看的电动车" → 是"drive Ferrari的仪式"
- 四圆灯传承 → drive行为的视觉确认

Sailing Lantern：
- 不是"好看的户外灯" → 是"extinguish/carry/hang的光"
- 单旋钮设计 → 行为统一控制
```

### 应用模板

```
Step 1: 列出产品的核心行为
├── 这个产品让用户能做什么？
├── 这些动词是什么？
└── 有没有更精确的动词？

Step 2: 改写为行为动词
├── "关灯" → "extinguish"
├── "拿东西" → "carry"
└── "听音乐" → "listen"

Step 3: 基于行为动词设计形态
├── extinguish → 需要熄灭动作
├── carry → 需要悬挂/背负结构
└── listen → 需要与唱片机的仪式关系
```

## 方法2: 先重写关系，再重写形态

### 核心原理 [S][A]

```
关系命题模板：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
This is not a [default category];
it is a relationship between [person],
[action], [environment], and [trust].
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

核心问题：
┌────────────────────────────────────────────┐
│                                            │
│  这个产品不是"某物"，                      │
│  而是"什么人" 在 "什么环境"               │
│  通过 "什么动作" 建立 "什么信任"          │
│                                            │
└────────────────────────────────────────────┘
```

### 深度理解：为什么"先重写关系"？

```
表面理解：
→ 先写关系命题，再做设计

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么？
→ 默认品类答案会限制思考
→ "台灯" → 想到台灯造型 → 无法突破
→ 先重写关系 → 重新定义问题 → 形态创新

本质是什么？
→ 设计是定义问题的能力，不是解决问题的能力
→ 正确的问题 → 正确的解决方案
→ 错误的问题 → 无论方案多好看都错

怎么做？
→ 不接受"这是一个XX产品"的说法
→ 问："这不是XX，那这是什么关系？"
→ 重新定义关系 → 形态自然浮现
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Sailing Lantern：
- 默认答案："好看的户外灯"
- 重写关系："人与光、携带、熄灭、海洋环境的关系"
- 形态推导：织带、Fresnel透镜、单旋钮

Ferrari Luce：
- 默认答案："电动法拉利"
- 重写关系："人与Ferrari EV架构、驾驶仪式的关系"
- 形态推导：长车头、四圆灯、物理控制
```

## 方法3: 熟悉但非怀旧

### 核心原理 [S][A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   熟悉感来自人类已经理解的动作和秩序                     │
│   不是复制旧时代风格符号                                │
│                                                         │
│   可以引用：              必须附：                       │
│   ├─ archetype          └─ not nostalgic replica       │
│   ├─ ritual             └─ not retro prop          │
│   ├─ ritual             └─ not decorative pastiche   │
│   └─ ceremonial seal                                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：什么是"熟悉"？

```
表面理解：
→ 用一些旧时代元素但不要太像

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
什么是"熟悉"？
→ 不是视觉上的相似
→ 是行为上的熟悉
→ 用户看到这个产品，不需要学习就会用

什么是"怀旧"？
→ 复制旧时代的外观
→ 假装技术停在过去
→ 没有真正理解为什么过去的东西是好的

为什么重要？
→ 人需要熟悉感来建立信任
→ 但怀旧会让人觉得产品在"假装"
→ 需要引用archetype，但用现代方式实现

怎么做？
→ 问："这个品类的archetype是什么？"
→ 问："这个archetype在解决什么问题？"
→ 问："用现代材料和工艺如何实现同样的解决方案？"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Sailing Lantern：
- 引用：航海灯的秩序（圆形、玻璃、金属）
- 但用：LED + Fresnel + 不锈钢
- 行为相同，材料更新

Ferrari Luce：
- 引用：四圆灯的品牌记忆
- 但用：纯电架构 + 全新比例
- 仪式相同，技术更新
```

### 案例解析

**Sailing Lantern**：
- 引用航海灯记忆 ✓
- 但用 LED + Fresnel + 不锈钢实现 ✓
- 不是复制 1950 年代航海灯 ✗

**Linn LP12-50**：
- 引用经典唱盘架构 ✓
- 但用现代金属铰链 + 数字控制 ✓
- 不是复制 1970 年代木盒 ✗

**Ferrari Luce**：
- 引用 Ferrari 四圆灯 + 驾驶仪式 ✓
- 但在 EV 体量中重新组织 ✓
- 不是复制化石燃料 Ferrari ✗

### 检查清单

```
□ 这个引用是否引用了动作/秩序，而非外形？
□ 是否有 not nostalgic replica / not retro prop 约束？
□ 引用是否服务于当前关系命题？
□ 还是只是在复制旧时代符号？
```

## 方法4: 边界即尊重

### 核心原理 [S]

```
边界不是限制，是设计的伦理边界

┌────────────────────────────────────────────┐
│                                            │
│  艾维不把以下视为限制：                     │
│  ├─ 工程约束                               │
│  ├─ 法规要求                               │
│  ├─ 材料限制                               │
│  ├─ 维护条件                               │
│  └─ 公开事实                               │
│                                            │
│  而是视为：                                │
│  └─ 设计的伦理边界                         │
│                                            │
└────────────────────────────────────────────┘
```

### 深度理解：为什么"边界是尊重"？

```
表面理解：
→ 尊重边界，不要硬来

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么边界是"尊重"？
→ 边界是客观事实，不是主观限制
→ 不尊重边界 = 不尊重现实
→ 设计需要与现实合作，不是对抗现实

本质是什么？
→ 设计师的创造力不是"无视边界"
→ 而是"在边界内找到最优解"
→ 真正的创新是在约束下的解决方案

怎么做？
→ 不问"怎么突破这个边界"
→ 问"这个边界在告诉我什么？"
→ 把边界当作设计输入
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Sailing Lantern：
- 边界：IP67防水、耐盐雾、户外环境
- 尊重边界 → 不锈钢护架、圆端盖密封
- 边界成为设计决策的依据

Terra Carta Seal：
- 边界：制度符号的庄重要求
- 尊重边界 → 严格圆形秩序、珠点边界
- 边界成为克制设计的框架
```

### 边界清单模板

```
【不可谈判边界】
├── 环境边界
│   ├── 户外 → IP67 防水
│   ├── 海洋 → 耐盐腐蚀
│   └── 极端温度 → 材料耐候性
│
├── 法规边界
│   ├── 安全标准
│   ├── 认证要求
│   └── 进口限制
│
├── 材料边界
│   ├── 可获得性
│   ├── 成本约束
│   └── 加工难度
│
├── 维护边界
│   ├── 可维修性
│   ├── 备件供应
│   └── 用户自维护能力
│
└── 公开事实
    ├── 品牌 DNA
    ├── 已有产品线
    └── 技术限制
```

### 应用规则

```
规则 1: 边界缺失时，先输出问题，不生成外观
规则 2: 边界必须先于形态决策
规则 3: 违反边界的形态，无论多好看，都应拒绝
```

## 方法5: 材料诚实

### 核心原理 [S][A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   材料 = 功能 + 情绪 + 时间 的证据                      │
│                                                         │
│   不是：                                                │
│   ❌ premium metal                                     │
│   ❌ luxury wood                                       │
│   ❌ crystal clear                                     │
│   ❌ high-end finish                                   │
│   ❌ elegant texture                                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：什么是"材料职责"？

```
表面理解：
→ 不要用"高级感"这种词

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么不能只说"高级"？
→ "高级"是主观感受，无法验证
→ 不同人对"高级"的理解不同
→ 无法指导设计决策

什么是"材料职责"？
→ 每个材料在这个产品中承担什么？
→ 功能职责：物理/化学/生物功能
→ 情绪职责：传达什么感受
→ 时间职责：十年后变成什么样

怎么做？
→ 每个材料都要回答三个问题：
  1. 这个材料在这里做什么功能？
  2. 这个材料传达什么情绪？
  3. 这个材料十年后会变成什么样？
```

## 方法6: 触觉优先

### 核心原理 [S][A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   触觉 = 信任的媒介                                     │
│                                                         │
│   重要动作需要：                                        │
│   ├─ 阻尼 (damping)                                    │
│   ├─ 位置 (position)                                   │
│   ├─ 声音 (sound)                                      │
│   ├─ 状态点 (state indicators)                         │
│   └─ 可见边界 (visible limits)                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：为什么"触觉是信任"？

```
表面理解：
→ 要注意产品的触感

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么触觉是"信任"的媒介？
→ 手是最诚实的感官
→ 眼睛可以被欺骗，但手会告诉你真相
→ 用户拿起产品的那一刻，触觉建立了第一信任

本质是什么？
→ 触觉是最直接的身体确认
→ 用户通过触觉知道产品在"说真话"
→ 视觉可以假装，但触觉不能

为什么重要？
→ 触觉节点是高频交互点
→ 用户会反复触摸这些地方
→ 触觉决定了长期使用体验

怎么做？
→ 问："用户的手在哪里确认状态？"
→ 问："每个触觉节点的反馈是什么？"
→ 确保阻尼/位置/声音/状态点都到位
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Apple Watch表冠：
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  触觉节点：表冠                                         │
│  承担行为：滚动/缩放/选择                               │
│  反馈形式：旋转阻尼 + 按压确认                          │
│                                                         │
│  触觉节点：侧边按钮                                     │
│  承担行为：电源/Apple Pay                               │
│  反馈形式：按压确认                                     │
│                                                         │
│  触觉节点：表带连接                                     │
│  承担行为：佩戴/更换                                    │
│  反馈形式：磁吸锁定声                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 触觉节点清单

```
每个产品必须包含至少 3 个触觉节点：

1. 控制类 (Control)
   ├─ 旋钮 (dial)
   ├─ 按钮 (button)
   ├─ 拨杆 (switch)
   └─ 滑块 (slider)

2. 握持类 (Grip)
   ├─ 手柄 (handle)
   ├─ 边缘 (edge)
   └─ 表面 (surface texture)

3. 连接类 (Connection)
   ├─ 铰链 (hinge)
   ├─ 磁吸 (magnetic)
   ├─ 卡扣 (clip)
   └─ 织带 (strap)

4. 确认类 (Confirmation)
   ├─ 压印 (emboss)
   ├─ 凸起 (bump)
   ├─ 凹槽 (groove)
   └─ 铭牌 (badge)
```

### 触觉节点设计规则

```
每个触觉节点必须回答：
┌────────────────────────────────────────────┐
│                                            │

│  A:  亮度调节、色温切换、开合角度...       │
│                                            │
│  Q2: 阻尼/反馈是什么？                     │
│  A:  咔嗒声、段落感、顺滑...              │
│                                            │
│  Q3: 位置/形状如何帮助识别？               │
│  A:  凸起位置、刻度、纹理...              │
│                                            │
│  Q4: 声音/触感传达什么状态？               │
│  A:  开启/关闭、电量、连接...             │
│                                            │
└────────────────────────────────────────────┘
```

### 深度理解：什么是"十年后的信任"？

```
表面理解：
→ 产品要能维修

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

│                                                         │
└─────────────────────────────────────────────────────────┘
```

```
├── 维修工具是否通用？
```

## 方法8: 小构件大命题

### 核心原理 [A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   真正重要的设计动作往往发生在                             │
│   每天被触摸，但最容易被忽略的地方                        │
│                                                         │
│   成熟产品不一定需要重画外形                              │
│   改变集中在关键接触点                                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：为什么"小构件"重要？

```
表面理解：
→ 要注意小细节

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么小构件承担"大命题"？
→ 用户每天触摸的是小构件
→ 小构件的体验决定了整体体验
→ 小构件是触觉信任的关键点

本质是什么？
→ 改变小构件可以解决大问题
→ 不需要推翻整体，只需要升级关键接触点
→ 小步改进 vs 大换血

怎么做？
→ 识别"关键接触点"
→ 评估每个接触点是否承担重要行为
→ 判断是否需要大改还是小升级
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
LP12-50 vs LP12原版：
- 整体架构：未推翻 ✓
- 保留：透明亚克力盖、黑胶垫、唱臂基础结构

改变的接触点：
├─ 铰链 → 金属铰链（功能+美学升级）
├─ 按钮 → 金属实心按钮（触觉升级）
├─ 底座 → Bedrok plinth（声学升级）
└─ 铭牌 → 新lid badge（身份升级）
```

### 关键接触点清单

```
遇到经典产品再设计时：

Step 1: 列出所有"关键接触点"
├── 旋钮 / 按钮 / 拨杆
├── 铰链 / 开关 / 连接件
├── 织带 / 手柄 / 握持点
├── 压印 / 铭牌 / 标识
└── 盖板 / 接缝 / 边界

Step 2: 评估每个接触点
├── 这个接触点改变了是否影响用户信任？
├── 这个接触点是否承担了重要行为？
├── 这个接触点是否在老化/维护路径上？

Step 3: 决定是否改变总体轮廓
├── 如果关键接触点需要大改 → 考虑重画外形
├── 如果关键接触点可以小改 → 保持轮廓，只升级接触点
└── 禁止为了显得新而强行重构品类
```

### 深度理解：什么是"秩序"？

```
表面理解：
→ 要有统一的设计语言

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 方法11: 反屏幕主义是约束，不是风格

### 核心原理 [S][A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   减少屏幕中心化 ≠ 删除屏幕                             │
│                                                         │
│   = 避免让所有行为都被一块玻璃吞没                       │
│                                                         │
│   distributed physical and ambient interface             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：为什么"反屏幕"？

```
表面理解：
→ 不要太多屏幕

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么屏幕会变成问题？
→ 屏幕可以承载任何功能
→ 结果：所有功能都被塞进屏幕
→ 人的注意力被屏幕绑架

本质是什么？
→ 屏幕适合：信息展示、复杂导航
→ 屏幕不适合：高频交互、状态确认
→ 物理界面适合：高频交互、即时反馈

为什么重要？
→ 高频交互需要身体确认
→ 屏幕无法提供触觉反馈
→ 长期使用后，物理界面的效率更高

怎么做？
→ 问："这个操作需要身体确认吗？"
→ 需要 → 物理界面
→ 不需要 → 屏幕
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Apple Watch：
- 屏幕：信息展示
- 表冠：高频控制（滚动/缩放）
- 侧键：快捷功能
- 分离：高频vs低频

Ferrari Luce：
- 推断：分布式物理控制
- 不是 touchscreen-only
- 方向盘、仪表、旋钮各有分工
```

### 界面分配原则

```
问：这个操作需要身体确认吗？

需要身体确认（→ 物理界面）：
├─ 高频交互（调节亮度、音量）
├─ 安全相关（启动、停止）
├─ 状态确认（电量、连接）
└─ 仪式动作（播放、暂停）

不需要身体确认（→ 屏幕）：
├─ 信息展示
├─ 详细设置
├─ 导航浏览
└─ 数据可视化
```

### 深度理解：为什么"摄影分工"？

```

深度理解：

→ 不是展示，是验证判断是否正确
→ 缺失某个角度 = 无法验证某个判断

怎么做？
→ 按四类角度组织图像证据
→ 每个角度回答一个设计问题
→ 缺失角度要明确标注
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
四类图像对应四类判断：
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Hero View → 关系命题是否正确？                        │
│  Side View → 比例是否和谐？                            │
│  Macro View → 触觉节点是否到位？                       │
│  Use-state → 真实使用体验如何？                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 四类摄影角度

```
1. Hero View（整体判断）
   问题：整体关系是什么？是否正确？
   
2. Side Proportion（比例判断）
   问题：各部分比例是否和谐？是否符合关系命题？
   
3. Macro Tactile（触觉判断）
   问题：触觉节点是否清晰？材料和工艺是否诚实？
   
4. Use-state（场景判断）
   问题：在真实使用中，这个产品如何存在？
```

### 应用规则

```
规则 1: 某类证据缺失时，标为"待补图"
规则 2: 不假装已经看到未拍的角度
规则 3: 不同角度回答不同问题，不能互相替代
规则 4: 最终判断需要四类证据齐全
```

## 方法13: 模块化身体关系

### 核心原理 [A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   主体 = 秩序层                                         │
│   接口 = 身体层                                         │
│                                                         │
│   两者用不同材料、不同节奏、不同生命周期管理             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：什么是"主体+接口"？

```
表面理解：
→ 产品要模块化

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么主体和接口要分离？
→ 主体需要稳定，接口需要灵活
→ 主体变化慢，接口变化快
→ 分离后，接口可以迭代而不影响主体

本质是什么？
→ 主体 = 不变的核心关系
→ 接口 = 变化的使用方式
→ 好的设计让接口多样化，同时保持主体稳定

怎么做？
→ 识别什么是"核心关系"（稳定）
→ 识别什么是"使用方式"（可变）
→ 用不同材料/节奏管理两者
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Apple Watch：
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  主体（表壳）                                           │
│  └─ 秩序层：计算核心、显示、人机界面                    │
│  └─ 变化节奏：慢（几年更新一次）                       │
│                                                         │
│  接口（表带）                                           │
│  └─ 身体层：材质、舒适、个性化                         │
│  └─ 变化节奏：快（随场合/季节更换）                    │
│                                                         │
│  分离管理：                                             │
│  └─ 主体保持品牌一致性                                  │
│  └─ 接口允许用户个性化                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 案例分析

| 产品 | 主体 | 接口 |
|------|------|------|
| Apple Watch | 表壳（秩序） | 表带（身体） |
| Moncler 外套 | 核心保暖层 | 磁性 Duo 按钮（连接） |
| Linn LP12-50 | 唱盘基础（声学秩序） | 升级件（可迭代） |
| Mac Pro 2019 | 主机（模块化架构） | PCIe 扩展（功能） |

## 方法14: 公共图形系统作为设计对象

### 核心原理 [A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   设计 ≠ 图形                                           │
│   设计 = 图形被使用的秩序                               │
│                                                         │
│   克制 = 可被规范化的方法                                │
│   不是美学口号                                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：为什么"规范"是设计？

```
表面理解：
→ 要制定图形使用规范

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么"规范"是设计的一部分？
→ 图形离开规范就不完整
→ 好的规范让图形在各种场景下都正确使用
→ 规范是"如何正确使用"的设计

本质是什么？
→ 克制不是"少做"
→ 克制是"做规则"
→ 规则越详细，执行越准确

怎么做？
→ 不只设计图形本身
→ 同时设计：clear zone、禁止变形、字体、缩放规则
→ 把"不应该做什么"也明确化
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
Terra Carta Seal规范：
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  图形本身：                                             │
│  ├─ 4国植物 × St Edward's Crown几何                   │
│  └─ 花卉是结构节点，不是装饰                           │
│                                                         │
│  使用规范：                                             │
│  ├─ clear zone规范                                    │
│  ├─ 禁止变形                                          │
│  ├─ 单一字样（LoveFrom Serif）                        │
│  └─ 使用场景规则                                       │
│                                                         │
│  克制 = 每个动作都被规则化 → 规则越多越稳定            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 规范层清单

```
公共图形 prompt 必须同时输出：

1. 图形本身
   └─ 形状、颜色、比例

2. 使用规范
   ├─ clear zone（最小空白）
   ├─ 禁止变形规则
   ├─ 单一字样（LoveFrom Serif）
   ├─ 缩放比例
   └─ 禁止项

只输出图形不输出规范 → 不完整
```

### 深度理解：为什么"组织"是设计对象？

```
表面理解：

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- 不是"LoveFrom给Airbnb做了设计"

- LoveFrom离开后，设计能力持续
```

```
协作类命题必须考虑：

1. 信任设计

   ├─ 信任如何通过触觉传达？

   ├─ 如何设计告知/提醒？
   ├─ 如何设计异常处理？
   └─ 如何设计确认机制？

3. 公共体验
   ├─ 公共接触点是什么？
   ├─ 如何统一体验？
   └─ 如何保持克制？

4. 视觉系统
   ├─ 图形规范
   ├─ 材料系统
   └─ 触觉语言
```

## 方法16: 消失型硬件与装配动作作为体验

### 核心原理 [S][A]

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   消失 = 让出注意力                                     │
│   不是目标，是结果                                       │
│                                                         │
│   被让出的注意力去向：                                   │
│   ├─ 身体                                              │
│   ├─ 仪式                                              │
│   ├─ 内容                                              │
│   └─ 信任                                              │
│                                                         │
│   装配 / 戴用 / 启动都是体验动作                         │
│   不是次要动作                                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 深度理解：什么是"消失"？

```
表面理解：
→ 产品要简洁

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么"消失"不是目标？
→ 消失是让注意力去到更重要的地方
→ 目标不是"产品消失"，而是"注意力被正确分配"

本质是什么？
→ 产品是手段，不是目的
→ 人的注意力是稀缺资源
→ 好的设计把注意力引向任务/内容

被让出的注意力去向：
→ 身体 → 舒适、自然
→ 仪式 → 使用过程的庄重感
→ 内容 → 音乐、视频、对话
→ 信任 → 对产品可靠性的确认

怎么做？
→ 问："这个产品让人的注意力在哪里？"
→ 如果注意力在产品上 → 太多
→ 如果注意力在任务/内容上 → 正确
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【视觉语料验证】[A]
AirPods：
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  被消失的是什么？→ 对"设备"的注意力                     │
│  让给的是什么？→ 音乐                                    │
│  怎么做到的？                                           │
│  ├─ 去除线缆 → 仪式感                                  │
│  ├─ 白色设计 → 品牌识别                                │
│  ├─ 轻量化 → 佩戴舒适                                  │
│  └─ 充电盒 → 续航保证                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘

Apple Park：
- 被消失的是什么？→ 对"建筑"的注意力
- 让给的是什么？→ 协作、自然光
```

### 检查清单

```
消失型 prompt 必须先定义：

1. 被让出的是什么？
   ├─ AirPods → 让出对"设备"的注意力给音乐
   ├─ Apple Park → 让出对"建筑"的注意力给协作
   └─ iPhone X → 让出对"手机"的注意力给内容

2. 留下的最小接口是什么？
   ├─ AirPods → 充电盒（公共身份）
   ├─ Apple Park → 庭院（协作触发）
   └── iPhone X → Face ID（解锁确认）

禁止：
└─ 只写 invisible / seamless / minimal
```

**方法 10 的"邻里式协作"补充** [A]：
```
"rising tides raise all boats"（水涨船高）的具体落地：
→ LoveFrom 在 Jackson Square 投 $90M 房地产，主动降租
→ LoveFrom 给街区其他商户（Quince 三星米其林）提供无偿设计服务
→ 整片街区变好，单个客户也跟着变好
→ 这是系统级协作，不是项目级协作
```

**艾维原话** [A]（关于 Stout Books 的发现）：
> "I discovered Stout Books on my first visit to San Francisco back in 1989. I have such a deep affection for this bookshop and the extraordinary community that Bill lovingly created over five decades."

**Chris Wilson（LoveFrom）[A]**：
> "It's a store we loved. And if we didn't get to design [their brand], it would have been like, 'we wish we could have done that!' It's something that falls into that category of projects we do for the love of doing them—quite organic by just being in the neighborhood and going in the store."

### 方法14（公共图形系统）· 新增证据：William Stout 完整规范系统 [A]

**Stout Books 公共图形系统的 5 维** [A]：
```
┌─────────────────────────────────────────────────────────────────────┐
│           William Stout 公共图形系统 · 完整可规范化                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 自研字体：LF Washington（跨平台 = 印刷 + 数字）                 │
│  2. 组合 logo：店名 + 始建日期（"Stout Books" + "Est. 1974"）       │
│  3. 配色：黑/白/红（基于 Prouvé + Le Corbusier 建筑传统）           │
│  4. 插画系统：Satoshi Hashimoto（含 Eames House Bird 彩蛋）         │
│  5. 标牌系统：现代版 LF Washington 用于所有印刷/数字资产            │
│                                                                     │
│  规 范 化 边 界：                                                    │
│  → "尊重 50 年" = 不允许重新设计 Stout 50 年档案里的元素           │
│  → "未来 50 年" = 自研系统要能用 50 年                             │
│  → clear zone / 禁止变形 / 单一字样规范（方法 14 标准）             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**方法 14 模板新增**：
```
公共图形系统 = 字体（自研）+ logo（组合命名）+ 配色（历史色板）
            + 插画（可识别 IP）+ 标牌（统一规范）
            + 50 年档案边界（不可重新设计）

5 维系统才能"克制 = 可被规范化"
```
