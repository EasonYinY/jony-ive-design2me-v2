## 2. LoveFrom 协作哲学

### 三大原则 [S][A]

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LoveFrom 协作三原则                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  原则1: 进入系统，而非外包造型                                    │
│  ──────────────────────────────────────────────────────────────     │
│  ❌ 外包模式：发brief → 交付设计 → 结束                           │
│  ✓ LoveFrom：进入系统 → 共同创造 → 能力转移                       │
│                                                                     │
│  原则2: 长期驻场，而非项目制                                      │
│  ──────────────────────────────────────────────────────────────     │
│  Airbnb: 4年驻场                                                   │
│  Linn: 5年合作                                                     │
│  Ferrari: 持续合作                                                  │
│                                                                     │
│  原则3: 帮合作方建团队，而非替合作方做                            │
│  ──────────────────────────────────────────────────────────────     │
│  提升合作方的材料/触觉/体验能力                                     │
│  让合作方在LoveFrom离开后仍能持续                                  │
│  成功标志：合作方比之前更强                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 深度理解：为什么"进入系统"？

```
表面理解：
→ 要多和合作方沟通

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么"外包"是错的？
→ 外包 = 交易关系
→ 外包 = 给brief，等结果
→ 外包 = 能力留在外部

为什么"进入系统"是对的？
→ 进入 = 合作关系
→ 进入 = 成为团队一部分
→ 进入 = 能力转移给合作方

本质是什么？
→ 真正的协作需要时间和物理接近
→ 了解对方的约束、不能失去的东西
→ 不是替对方做，是帮对方变强

艾维说 [S]：
> "我们不是去给一个brief，然后拿一个设计回来。我们是进入他们的组织。"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 合作前评估清单

```
在开始协作前，必须回答：[A]

1. 合作方是谁？
   ├── 品牌历史
   ├── 核心价值
   ├── 市场定位
   └── 已有能力

2. 合作方已经知道什么？
   ├── 工程能力
   ├── 制造能力
   ├── 品牌资产
   └── 用户理解

3. 合作方擅长什么？
   ├── 核心优势
   └── 差异化能力

4. 合作方不能失去什么？
   ├── 品牌DNA
   └── 核心用户

5. 合作方需要被提升什么？
   ├── 材料精度
   ├── 触觉语言
   └── 长期主义
```

## 3. 团队建设模式

### Apple设计团队哲学 [A]

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Apple设计团队管理                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【团队规模】                                                      │
│  小团队，精选择（20-30人）                                        │
│  → 不是要很多人，是要对的人                                        │
│                                                                     │
│  【决策机制】                                                      │
│  亲自参与每个关键决策                                              │
│  → 不是授权，是确保判断标准一致                                    │
│                                                                     │
│  【说服方式】                                                      │
│  用作品说话，而非用语言                                            │
│  → 不是开会讨论，是做出来看                                        │
│                                                                     │
│  【容错态度】                                                      │
│  容忍失败，快速迭代                                               │
│  → 失败是学习机会                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 深度理解：为什么"用作品说话"？

```
表面理解：
→ 不要只说不做

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么语言不如作品有效？
→ 语言可以被辩论
→ 作品可以被体验
→ 体验比辩论更有说服力

本质是什么？
→ 设计判断是感性的
→ 作品直接作用于感性
→ 语言在感性面前苍白无力

艾维的实践 [B]：
→ 不花时间写设计说明
→ 花时间做原型
→ 让乔布斯和团队体验原型
→ 体验后，判断自然形成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 团队管理特点 [A]

1. **以身作则**
   - 不是"设计总监签字"
   - 而是"每个细节都要过艾维"

2. **用作品说话**
   - 少开会，多做
   - 方案代替争论

3. **容忍失败**
   - 快速原型，快速失败
   - 从失败中学习

### 深度理解：什么是"在压力下保护团队"？

```
表面理解：
→ 老板要罩着团队

深度理解：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
保护什么？
→ 不是保护团队不做难的工作
→ 而是保护团队的判断标准

怎么保护？
→ 在商业压力面前，坚持正确的判断
→ 替团队挡掉不合理的干预
→ 给团队足够的时间做到正确

艾维的做法 [B]：
→ 乔布斯施加压力时，艾维用专业性来说服
→ 不是硬顶，是找到共同的"好"的标准
→ 最终让乔布斯认可艾维的判断
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 争议处理 [A]

| 争议方 | 处理方式 |
|--------|----------|
| 团队内部 | 用原型/模型展示，让作品说话 |
| 商业部门 | 用数据支持，从不用"我是设计师"压人 |
| 合作方 | 理解约束，找到共同目标，用专业性赢得尊重 |

## 4.5 LoveFrom 团队仪式（Stripe Sessions 2025 新增） [A]

> **来源**：2025-05-07 Stripe Sessions 公开对谈（Patrick Collison 主持，1 小时）
> **等级**：A（Ive 本人公开访谈原文 + 8 篇中文/繁体转写印证）

```
┌─────────────────────────────────────────────────────────────────────┐
│             LoveFrom 内部团队仪式（每周循环）                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【仪式 1：每周团队共餐】                                           │
│  设计师轮流为大家做饭                                               │
│  → 把"照顾彼此"内化成团队默认动作                                  │
│                                                                     │
│  【仪式 2：去彼此家做 design review】                                │
│  打破工作室边界，走进对方生活                                       │
│  → 在"非工作空间"里做判断                                          │
│                                                                     │
│  【仪式 3：跨学科小组】                                             │
│  团队 = 设计师 + 音乐家 + 字体设计师                               │
│  → 不只做"造型"，也做"声音"和"字形"                                 │
│                                                                     │
│  【仪式 4：Make things for each other】                              │
│  → 把"互相照顾"沉淀进产品（包装、UI、声音）                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**艾维原话** [A]（Stripe Sessions 2025）：
> "每周五早上团队轮流为大家做早餐。"
> "为彼此做设计"（make things for each other）
> "一个包装盒中的拉片设计，关系到数百万用户的体验。"

**这段话说明了什么** [A]：
```
为什么是"仪式"而不是"流程"？
→ 流程可以外包
→ 仪式只能在团队内部成立
→ 仪式是隐性判断的具象化
→ 仪式传递的是"判断标准"，不是"做事步骤"

为什么"做饭"和"做 design review"是一回事？
→ 都是"在场"
→ 都是在非正式场景里观察彼此
→ 都在建立"我们怎么对人对事"的共识
```

**对应 16 法的填充价值**：
- 方法 1（语言先行）："sincerely elevate the species"（真诚地提升人类）= LoveFrom 标志性词汇
- 方法 2（关系先行）："express gratitude to the species"（向物种表达感恩）= 把"用户"框定为"人-物种"关系
- 方法 10（协作本身是设计对象）：每周共餐 + 去彼此家 = 协作的具体形态

## 4.6 长期合作的"5 年 / 4 年"形态 [A][B]

> **来源**：Moncler × LoveFrom（2024-09）+ Airbnb × LoveFrom（2020+）+ Linn Sondek LP12-50（2023）

| 合作 | 时长 | 起始关系 | 一句关键引语 |
|------|------|----------|---------|
| Moncler | 5 年 | Ruffini 自述"像回到大学跟大师学习" | "I felt like I was going back to university to study with the master." |
| Airbnb | 4 年 | 驻场建团队，不交付造型 | — |
| Linn | 5 年 | "true collaboration" | — |

**Ruffini（Moncler CEO）[A]**：
> "I felt like I was going back to university to study with the master."

**艾维 [A]**（关于 Moncler 合作）：
> 我们用"collaboration"框定（不是"客户/乙方"），button 内侧印 LoveFrom logo、外侧印 Moncler logo（双向品牌 = 真正的协作关系）。

**5 年到底在做什么** [A]：
```

5 年换来的"奢侈"：
→ 没有"快速交付"的爆款压力
→ 允许"5 年只研究一个 button"
→ 允许"5 年只重新组织一块布"

成功标志：
→ Moncler 5 年后，Moncler 团队自己学会了"用 kimono 折叠方式成形"
→ Airbnb 4 年后，Airbnb 有了自己的设计团队
```

## 4.7 三方 / 跨组织协作模式 [A]

> **来源**：William Stout × LoveFrom × Eames Institute（2025-05）
> **等级**：A（LoveFrom Chris Wilson + Antonio Cavedoni + EIIC Lauren Smith 三方直接引语）

```
┌─────────────────────────────────────────────────────────────────────┐
│            William Stout 案例 · 三方协作矩阵                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  主体：旧金山 Jackson Square 建筑书店，1974 创立                   │
│  所有权：Eames Institute of Infinite Curiosity（EIIC，2022 收购）   │
│  关系触发：LoveFrom 与 Stout 隔街（Ive 2019 选址 Jackson Square）  │
│                                                                     │
│  【三 方】                                                          │
│  ① LoveFrom（主设计）                                             │
│  ② Eames Institute（所有权方 + 文化传承）                          │
│  ③ Chips（网站重新设计）                                            │
│                                                                     │
│  【共 同 边 界】                                                    │
│  50 年档案（老 catalogue、map、gift card、stationery）             │
│  = 不可谈判的尊重对象                                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**Chris Wilson（LoveFrom 联合主创）[A]**：
> "我们很荣幸能帮助诠释这家书店光明的未来，创造一种视觉语言，反映对未来 50 年的乐观，同时尊重之前的 50 年。"

**Lauren Smith（EIIC chief experience officer）[A]**：
> "We don't want it to disappear."

**这些事在协作里发生了**：
- LoveFrom 在 Jackson Square 投 $90M 房地产，主动降租
- LoveFrom 给街区其他商户（三星米其林 Quince）提供无偿设计服务
- 关键引语（Fast Company 总结）："rising tides raise all boats"（水涨船高）

**"水涨船高"具体怎么落地** [A]：
```
为什么"非客户/乙方"关系有效？
→ 客户/乙方 = 交易 → 设计完成 → 关系结束
→ 邻居/共同街区 = 长期 → 设计只是关系的一环

LoveFrom 怎么"养"街区？
→ 不只服务一个客户
→ 主动把整片街区拉起来
→ 整片街区变好，单个客户也跟着变好
→ 这是系统级协作，不是项目级协作
```

## 6. 协作能力自评清单

### 协作前检查

```
□ 能理解并尊重合作方的品牌DNA吗？
□ 能在进入系统之前先充分了解系统吗？
□ 能区分"帮合作方做"和"帮合作方成长"吗？
□ 愿意为了正确的合作放弃错误的交易吗？
```

### 协作中检查

```
□ 能用作品而非语言说服他人吗？
□ 能在争议中保持专业和克制吗？
□ 能在坚持判断的同时保持关系吗？
□ 能把能力转移给合作方吗？
```

### 协作后检查

```
□ 合作方比之前更强了吗？
□ 合作方能独立持续吗？
□ 合作方愿意再次合作吗？
□ 有可传承的协作方法论吗？
```
