## MTH-002 先重写关系，再重写形态

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由两个独立 `S` 级案例支撑。
- 来源指针：source-index.md 中 AF-S-001 与 MON-S-001；references/core/design-methodology.md:1-7
- 适用条件：需要重新定义使用者动作或关系的任务
- 反例：不改变人、动作、环境与信任条件，只替换轮廓或表面。
- 禁止外推：不得复制参考案例的外形
- 运行规则：先写清使用者、动作、环境和信任条件，再允许进入形态推导。

## MTH-003 熟悉但非怀旧

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `SL-A-001` 与 `LP-S-002` 支撑。
- 来源指针：source-index.md 中 SL-A-001 与 LP-S-002；references/core/design-methodology.md:8-14
- 适用条件：有真实历史对象和用户行为可核对时
- 反例：用年代感色彩、装饰或历史轮廓代替对已有行为记忆的理解。
- 禁止外推：不得用复古装饰或老 Apple 外形代替行为证据
- 运行规则：保留用户已经理解的动作和秩序，但使结构、工程与当下条件重新对应。

## MTH-004 将边界转化为可解释的尊重

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由两个独立 `S` 级案例支撑。
- 来源指针：source-index.md 中 SL-S-002 与 LP-S-001；references/core/design-methodology.md:15-21
- 适用条件：边界有官方事实并可说明对象时
- 反例：把法规、环境、工程或信息缺失当作绕过对象，或用含糊的高端叙事掩盖。
- 禁止外推：不得把限量或高价自动解读为仪式或尊重
- 运行规则：每项边界都要写明它保护的人、环境、工艺、合作关系或事实准确性。

## MTH-005 让材料承担职责

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `LP-S-001` 与 `MON-S-002` 支撑。
- 来源指针：source-index.md 中 LP-S-001 与 MON-S-002；references/core/design-methodology.md:22-28
- 适用条件：材料性能制造方式和寿命有可验证责任时
- 反例：只写"高级金属"或"奢华木材"，却无法说明它为何适合结构、交互、制造与寿命。
- 禁止外推：不得因材料昂贵或金属外观归因艾维方法
- 运行规则：材料决策必须同时写明性能职责、制造方式、触感、老化与维护后果。
- 强制格式：材料名称 | 结构职责 | 触感职责 | 制造方式 | 寿命 | 维护方式
- 禁止：使用纯颜色词（如"深岩灰""香槟金"）替代材料职责说明。颜色词仅作为材料职责确定后的补充描述，不得作为设计决策依据。

## MTH-006 用触觉节点承载确认与信任

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `SL-S-002` 与 `MON-S-002` 支撑。
- 来源指针：source-index.md 中 SL-S-002 与 MON-S-002；references/core/design-methodology.md:29-35
- 适用条件：存在可观察操作动作和反馈的任务
- 反例：为了显得“有交互”而添加无状态、无功能或反馈含糊的按钮。
- 禁止外推：不得把所有物理控件都称为艾维式交互
- 运行规则：对每个高价值操作说明手势、阻尼、位置、声音或其他状态反馈，并允许根据任务删除物理控件。

## MTH-007 将维护、修理与老化纳入生命周期

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `SL-S-002` 与 `LP-S-002` 支撑。
- 来源指针：source-index.md 中 SL-S-002 与 LP-S-002；references/core/design-methodology.md:36-42
- 适用条件：长期使用可维护性或老化是真实约束时
- 反例：封闭式外观看似整洁，但清洁、拆解、更换或回收路径不存在。
- 禁止外推：不得对所有品类强制十年寿命或指定材料
- 运行规则：先确认真实服役期和失效模式，再定义清洁、拆解、修理、更换、回收和可接受老化。

## MTH-008 让高频小构件承载核心命题

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `LP-S-002` 与 `MON-S-002` 支撑。
- 来源指针：source-index.md 中 LP-S-002 与 MON-S-002；references/core/design-methodology.md:43-49
- 适用条件：小构件对高频行为或系统状态有可验证影响时
- 反例：为了展示“全新”而重画整体，却保留日常最常触摸的粗糙节点。
- 禁止外推：不得把旋钮按钮或铰链变成通用外形符号
- 运行规则：先排序高频接触点和状态切换点，只在其中选择能实质改善体验的构件深入设计。

## MTH-009 用可解释的秩序收束复杂

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `TC-S-003` 与 `CE-S-002` 支撑。
- 来源指针：source-index.md 中 TC-S-003 与 CE-S-002；references/core/design-methodology.md:50-56
- 适用条件：功能或公共符号复杂但需要系统规范时
- 反例：为追求“极简”删除必要功能、语义或公共识别线索。
- 禁止外推：不得把极简当作唯一正确结果
- 运行规则：为每个必要元素分配职责、层级、边界与冲突规则，删除的是无职责元素。

## MTH-010 将协作关系与能力建设视为设计对象

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `AF-S-001` 与 `OI-S-001` 支撑。
- 来源指针：source-index.md 中 AF-S-001 与 OI-S-001；references/core/design-methodology.md:57-63
- 适用条件：服务系统组织或多团队产品
- 反例：把合作方当作空白画布，交付一次性方案却没有留下可运行的能力。
- 禁止外推：不得把 LoveFrom 的合作写成主导或外包
- 运行规则：先写清合作方已有能力、不可失去的身份、决策责任和需被增强的能力。

## MTH-014 将公共图形与其使用规范一同设计

- 证据类型：`INFERENCE`，跨案例推断。
- 证据等级：`A`，由 `CE-S-002` 与 `TC-S-003` 支撑。
- 来源指针：source-index.md 中 CE-S-002 与 TC-S-003；references/core/design-methodology.md:64-70
- 适用条件：标志需跨载体和公共场景稳定使用时
- 反例：只交付一张标志图，没有最小尺寸、留白、色彩、载体和禁止变形规则。
- 禁止外推：不得将具体皇室或 Terra Carta 纹样复制到新任务
- 运行规则：图形、字样、最小尺寸、安全留白、色彩、应用与禁止项属于同一份设计产物。
