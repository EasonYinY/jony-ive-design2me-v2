# Design IR 完整合同

## 用途

`DesignIR` 是产品判断与视觉表达之间的不可变中间表示。它只记录已由关系、约束、证据和五维判断推导出的产品定义；摄影、渲染风格和提示词不能反向修改它。

## 必填字段

| 字段 | 中文含义 | 完整性要求 |
|---|---|---|
| `direction_id` | 方向标识 | 与所属 `DirectionBundle` 完全一致 |
| `relationship` | 人与产品的关系 | 写清使用者、产品角色、环境、控制权与信任条件 |
| `structure` | 结构 | 写清承力、构件布置、连接、开合和模块边界 |
| `operation` | 操作方式 | 写清关键动作、顺序、状态切换和完成条件 |
| `form_proportion` | 形态比例 | 写清主次体量、轮廓转折、接缝秩序、部件层级及其结构依据 |
| `cmf_system` | CMF 系统 | 写清主辅材、颜色、成形与表面工艺、触感、老化和回收职责 |
| `resting_state` | 静息状态 | 写清停止使用或断电时的构件位置、稳定性、状态可读性与收纳关系 |
| `ergonomics` | 人体工学 | 写清姿态、握持、触达、视线、受力与可及性边界 |
| `materials` | 材料 | 写清性能职责、触感、老化、环境与回收后果 |
| `manufacturing` | 制造 | 写清成形、装配、公差、规模、质量控制和成本约束 |
| `interaction` | 交互 | 写清输入、反馈、系统状态、错误预防和恢复路径 |
| `maintenance` | 维护 | 写清清洁、拆解、诊断、修理、更换和回收路径 |
| `distinctive_features` | 识别特征 | 至少一项，由关系、结构或操作自然导出，不得是风格贴花 |

所有字段必须为非空文本；`distinctive_features` 必须是非空且不可变的文本元组。

## 形成步骤

1. 输入：读取本方向自己的输入、证据、候选与否决理由。
2. 核心问题：确认该方向要建立的关系命题，而非外观主题。
3. 五维判断：逐维检查价值、系统、可行性、执行和时机。
4. 证据：只采用有来源指针且符合适用条件的事实、跨案例推断或治理规则。
5. 候选：比较本方向内部候选，不借用其他方向的结论。
6. 否决理由：记录候选为何违反约束、证据、人体工学、制造或维护要求。
7. 结论：填写九类产品定义和识别特征。
8. 未确认项：保留需计算、调研、样机或用户测试的事项，不伪装成已确认规格。
9. 状态：字段齐全不代表事实已验证；只有方向协议校验通过才进入 `IR_READY`。
10. 下一步：N07 只允许停在 `IR_READY`，不得提前进入提示词编译。

## 一致性与否决规则

- 关系必须能解释结构、操作、形态比例与静息状态；无法解释时退回 `EXPLORING`。
- `form_proportion` 必须由身体、受力、构件布置与动作导出；`cmf_system` 必须与材料和制造字段一致；`resting_state` 必须能由结构和维护路径实现。
- 材料必须承担结构、触感、寿命或环境职责，不能只表达“高级感”。
- 制造必须说明可实现路径；无法制造的概念不能以视觉完整代替。
- 交互与人体工学必须包含可观察动作和反馈，不能只列控件。
- 维护必须与真实服役期和失效模式一致，不能默认封闭或永久寿命。
- 识别特征必须来自本方向的产品逻辑；颜色、纹理、品牌符号和 Apple-like 轮廓不合格。
- 未公开产品信息、无来源人物意图和待核验假设不得进入正向字段。

## 与三方向集合的关系

单个 `DesignIR` 完整后，仍必须与另外两个方向一起校验。三个 `relationship`、三个 `structure` 和三个 `operation` 必须分别唯一；任一重复均阻断整个集合进入下一阶段。
