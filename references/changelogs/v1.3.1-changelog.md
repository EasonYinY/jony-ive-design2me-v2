# v1.3.1 — 功能分区（Functional Zoning）方法论 + jony-ive↔lovart 协议钩子

> 用户原话（2026-06-23 咖啡机任务复盘）：
> - "中规中矩，很普通"
> - "功能比较复杂的产品需要做功能的分区，有多个功能区来组成，每个区域都要有内部的结构、功能"
> - "一个方块就全部概括了。简单的小东西可以尝试这么处理，但是不要所有的方案都这样去做"

## 触发根因

1. **STEP-06 候选生成单位错位**：把"1 个基础体块"当成"1 个方向"——5 个方向都是 1 个整体造型（slab/cylinder/block/wedge/dome），5 个功能区被压扁
2. **STEP-10 形态推导层级错位**：从"1 个体块"出发，没有"5 个功能区各自的基础体块 + 区间接缝"
3. **STEP-15 提示词结构错位**：单一段落描述整个产品，不是分段功能区描述
4. **vision 评审打分偏松**：按 DesignIR 一致性打 7-8/10，但按艾维级行业对照只有 5/10
5. **jony-ive ↔ Lovart 协议裂缝**：jony-ive SKILL.md §"Lovart 调用模式"只写了 `agent_skill.py chat` 没写后续必跑 `eagle_publish.sh`；Lovart skill v2.x 已有 Eagle 协议但没声明"被 jony-ive 自动调用"

## 关键决策

| 决策 | 理由 |
|---|---|
| 功能区分级触发（≥ 3 必做，≥ 5 必每区独立体块） | 简单产品（鼠标/剃须刀）1 个体块足够；复杂产品（咖啡机/无人机/医疗设备）必须拆区 |
| DesignIR 新增 3 字段而非合并到 structure | "功能区" 是和"结构"并行的概念，合并会模糊"功能区 = 视觉信号"的清晰度 |
| 视觉评审双评分（DesignIR 一致性 + 艾维级对照） | 单一评分会被"形式合规"误导；双评分让"形式合规但行业平庸"显式可见 |
| Lovart 钩子放 "When to Use" 而非"Default Post-Generation" | 显式声明"被 jony-ive 自动调用"比"默认行为"更易被未来 agent 检索到 |

## 协议裂缝反思

**为什么 2 个 skill 之间会有裂缝**：
- Lovart skill 升级时（v2.0.0）只声明了"被用户手动调用"
- jony-ive skill 升级时（v1.2.1）只 patch 了 Pitfall 024-025，没意识到 Lovart skill 已有 Eagle 协议
- **结论**：跨 skill 协议必须**双向声明**——"我能调用谁" + "谁会调用我"，否则任何一边的升级都会留下裂缝
- **下次 v1.3.2+ patch 规范**：跨 skill 协议改动必须同步修改两个 skill 的 SKILL.md
